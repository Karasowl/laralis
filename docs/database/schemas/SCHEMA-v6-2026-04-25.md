## What's New in v6

This version completes the WhatsApp mini-CRM data model by adding:

1. **Click-to-WhatsApp ad attribution** on the `leads` table â€” the webhook now captures `ctwa_clid`, `ad_id`, `ad_source_url`, `ad_headline`, `ad_body`, `ad_media_*` from the Twilio (and future 360dialog) referral payload of the first inbound message of each lead.
2. **Explicit `lead_id` foreign key on `treatments`** â€” closes the funnel CTWA â†’ conversation â†’ lead â†’ patient â†’ treatment with a direct join, instead of relying on the implicit chain through `converted_patient_id`.

Both changes are backwards compatible (all new columns nullable, all FKs `ON DELETE SET NULL`). A backfill is applied at migration time to link existing converted leads to their first treatment.

## Modified Tables

### `leads` (Migration 77)

New columns:

| Column | Type | Notes |
|---|---|---|
| `ctwa_clid` | TEXT | Click-to-WhatsApp click identifier from Meta. Joinable with Meta Ads conversion exports for revenue attribution. |
| `ad_id` | TEXT | Ad or post ID that generated the click (`referral.source_id`). |
| `ad_source_type` | VARCHAR(20) | Either `"ad"` or `"post"` (`referral.source_type`). |
| `ad_source_url` | TEXT | Source URL of the ad. |
| `ad_headline` | TEXT | Ad headline shown to the user. |
| `ad_body` | TEXT | Ad body text shown to the user. |
| `ad_media_type` | VARCHAR(20) | `image` / `video` / `none`. |
| `ad_media_url` | TEXT | Media URL of the ad creative. |

Indexes:
- `idx_leads_ctwa_clid` (partial, where `ctwa_clid IS NOT NULL`)
- `idx_leads_ad_id` (partial, where `ad_id IS NOT NULL`)

**Capture rule (first-touch wins)**: the webhook only writes these fields if the lead does not already have a `ctwa_clid`. A returning visitor who clicks a different ad does not overwrite the original attribution.

### `treatments` (Migration 77)

New column:

| Column | Type | Notes |
|---|---|---|
| `lead_id` | UUID | FK â†’ `leads.id`, `ON DELETE SET NULL`. Optional. |

Index:
- `idx_treatments_lead` (partial, where `lead_id IS NOT NULL`)

Backfill performed at migration time:

```sql
UPDATE treatments t
SET lead_id = l.id
FROM leads l
WHERE t.lead_id IS NULL
  AND l.converted_patient_id = t.patient_id
  AND l.clinic_id = t.clinic_id
  AND t.id = (
    SELECT id FROM treatments
    WHERE patient_id = l.converted_patient_id AND clinic_id = l.clinic_id
    ORDER BY treatment_date ASC NULLS LAST, created_at ASC LIMIT 1
  );
```

## Migration paths

To upgrade from v5 to v6 just run migration `77_lead_attribution_and_treatment_link.sql` in Supabase. Idempotent (uses `IF NOT EXISTS`).

To roll back: drop the new columns. The backfill is non-destructive â€” dropping the column simply loses the link, no data is destroyed.

---

## Compatibility notes

- The `leads.metadata` JSONB column is still available for non-CTWA referral payloads and other ad-hoc data.
- The webhook (`/api/whatsapp/webhook`) writes the full referral object into `inbox_messages.metadata.ctwa_referral` regardless of column population, so audit trail is preserved even if the lead row is updated later.
